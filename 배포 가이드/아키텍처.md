# EnglishAgentHub 배포 아키텍처

## 권장 구조

```text
사용자
  |
  | HTTPS
  v
english.dxline-tallent.com
  |
  v
CloudFront
  |------------------------|
  |                        |
  v                        v
S3 정적 프론트             EC2 Nginx
english-agent-hub-front    /api/* -> localhost:4301
                           |
                           v
                           Spring Boot
                           englishagenthub.service
                           |
                           v
                           PostgreSQL Docker
                           localhost:5436/english_agent_hub
```

## BeautyBook과의 공존 구조

english.dxline-tallent.com

## 포트 분리

| 서비스 | 포트 |
| --- | --- |
| BeautyBook Spring Boot | `4101` |
| BeautyBook PostgreSQL | `5434` |
| EnglishAgentHub Spring Boot | `4301` |
| EnglishAgentHub PostgreSQL | `5436` |
| Nginx | `80`, `443` |

## 프론트 배포 방식

EnglishAgentHub 프론트는 Next.js static export를 사용한다.

- `NODE_ENV=production`에서 `output: "export"` 활성화
- 빌드 산출물: `english-agent-hub-front/out`
- 권장 배포 대상: `s3://english-agent-hub-front`
- API 환경변수: `NEXT_PUBLIC_API_URL=https://english.dxline-tallent.com`

## 백엔드 배포 방식

- 로컬에서 Gradle 빌드
- JAR를 EC2의 `/home/ubuntu/english-agent-hub/app.jar`로 업로드
- systemd `englishagenthub.service`로 실행
- 환경변수는 `/home/ubuntu/english-agent-hub/.env`에서 로드

## S3 버킷 분리 여부

S3는 별도 버킷을 권장한다. EC2처럼 인스턴스 고정비가 붙는 구조가 아니고, 저장량/요청량 기준 과금이라 작은 정적 프론트 하나를 위한 버킷 추가 비용 부담은 거의 없다.

같은 버킷을 쓰려면 EnglishAgentHub를 `/english-agent-hub/` prefix에 올리고, Next.js `basePath`와 `assetPrefix`, CloudFront Function rewrite를 함께 맞춰야 한다. 단순하고 안전한 방식은 S3 버킷을 분리하고 EC2 백엔드만 공유하는 것이다.
